## Оригинальное ТЗ

### Описание проекта
Необходимо реализовать модуль Candidate Dashboard — дашборд для HR-специалиста, который позволяет просматривать список кандидатов, анализировать их соответствие вакансии и управлять статусами обработки резюме.

### Технические требования

Стек:
- React 18 + TypeScript
- Vite (сборка)
- Redux Toolkit или Zustand (state management)
- React Router (маршрутизация)
- Tailwind CSS или SCSS (стилизация)
- Jest + React Testing Library (тестирование)

API:
- Основной мок — mock/candidates.json (25 кандидатов, задания 1–4)
- Большой мок — mock/candidates-large.json (120 кандидатов, задание 5)

Структура кандидата:
```json
{
  "id": "ivanov",
  "name": "Иванов Иван Иванович",
  "position": "react-middle",
  "pos_label": "React — ведущий программист",
  "email": "ivanov@email.com",
```

### Задание 1. Список кандидатов
- Таблица: ФИО, город, опыт, вердикт (цветовая индикация), стек
- Фильтрация по вердикту: Все / ПОДХОДИТ / ЧАСТИЧНО / НЕ ПОДХОДИТ
- Поиск по ФИО с debounce 300ms
- Сортировка по имени, опыту, дате добавления
- Пагинация 10 кандидатов на странице
- Состояние фильтров сохраняется в URL (query params)
- Адаптивная вёрстка, обработка пустого результата

### Задание 2. Карточка кандидата
- Детальная страница /candidate/:id
- Полная информация: контакты, опыт, образование, стек
- Блок «Критерии оценки» с цветовой индикацией (ok/partial/no)
- Блок «Вопросы для собеседования»
- Блок «Summary»
- Кнопка «Назад к списку» с сохранением фильтров
- Обработка 404

### Задание 3. Управление статусами
- Выпадающий список: Новый → На рассмотрении → Приглашён → Отклонён
- Mock PATCH запрос: PATCH /api/candidates/:id/status
- Оптимистичное обновление с откатом при ошибке
- Loading state во время запроса
- Toast уведомления об успехе/ошибке

### Задание 4. Тестирование
- Unit: фильтрация по вердикту, поиск с debounce, цветовая индикация verdict
- Integration: переход список → детальная страница
- React Testing Library (не Enzyme)
- Моки API вызовов
- Edge cases: пустой список, ошибка загрузки

### Задание 5. Оптимизация (дополнительно)
- candidates-large.json (120 кандидатов)
- Виртуализация react-window
- useMemo для фильтрации/сортировки
- React.memo для тяжёлых компонентов
- Нет утечек памяти (cleanup в useEffect)

### Что оценивается

Обязательно:
- Чистота и читаемость кода
- Структура проекта и разделение ответственности
- TypeScript без any
- State management без prop drilling
- Обработка ошибок и loading states

## Мои заметки

### Принятые решения
- **Zustand** вместо Redux Toolkit — меньше бойлерплейта, современнее, ТЗ даёт выбор
- **createBrowserRouter** — современный подход v7, поддерживает data APIs
- **sonner** для toast — современная и лёгкая
- **Структура из ТЗ** (не FSD) — ТЗ явно указал структуру; знаком с FSD, упомяну в README
- **Zod не использую** — нет форм/реального API; знаком, упомяну в README
- **React Query не использую** — мок данные; знаком, упомяну в README

### Что писать в README
- Zustand: современнее, меньше бойлерплейта, ТЗ даёт выбор
- FSD: знаком, применил бы в крупном продакшен проекте. Здесь выбрал структуру из ТЗ
- Zod: знаком, применил бы для валидации форм и реальных API ответов
- React Query: знаком, применил бы для работы с реальным API
- Старт разработки: 12 июня 18:00 — видно по истории коммитов
