## Выбор инструментов

### Zustand вместо Redux Toolkit
ТЗ допускает оба. Выбрал Zustand: меньше бойлерплейта, нет провайдеров и экшен-криэйторов, удобные селекторы. Состояние разделено на два независимых стора — `candidatesStore` (данные + загрузка + смена статуса) и `filtersStore` (поиск, вердикт, сортировка, страница). Везде используются селекторы (`useStore(s => s.field)`), чтобы компоненты не перерисовывались на несвязанные изменения.

### React Router v7 — `createBrowserRouter`
Современный data-router API v7. Маршруты вынесены в отдельный модуль `routes/`.

### sonner для toast
Лёгкая и современная библиотека уведомлений, минимум настройки. Используется для обратной связи при смене статуса.

### react-window для виртуализации
Стабильная v1 с `FixedSizeList`. Строки фиксированной высоты → расчёт видимого окна без измерения DOM. Применена в отдельном режиме на 120 кандидатах.

### Tailwind CSS v4
Быстрая разработка консистентного UI. Утилита `cn()` (clsx + tailwind-merge) для безопасного объединения классов.

## Структура проекта

Использована простая структура из ТЗ (`components / pages / hooks / store / services / ...`), а **не FSD**.

**Про FSD:** активно работаю с этой методологией (слои `app / pages / widgets / features / entities / shared`, правило однонаправленных импортов) и применяю её в продакшене — на крупных проектах с несколькими командами и растущим числом фич она даёт жёсткую изоляцию слоёв. Для тестового задания такого объёма FSD был бы оверинжинирингом, поэтому осознанно выбрал структуру, заданную в ТЗ. Доменные сущности при этом вынесены отдельно (`constants/`, `types/`) как единый источник правды — это перекликается со слоем `entities`.

## Что намеренно не использовал

### Zod
Активно использую (схемы валидации, инференс типов из схем) для валидации форм и проверки данных с реального API на рантайме. В этом проекте нет ни форм со сложной валидацией, ни реального бэкенда — данные приходят из статичных моков и уже типизированы, поэтому рантайм-валидация была бы избыточной.

### React Query (TanStack Query)
Активно работаю с ним (кэширование запросов, инвалидация, фоновые рефетчи, оптимистичные апдейты «из коробки»). Применил бы здесь при работе с реальным API — это убрало бы ручное управление loading/error в сторе. Но данные мок-статичные и грузятся один раз, поэтому простого Zustand-стора с ручной загрузкой достаточно; тащить React Query ради одного запроса не оправдано.

## Оптимизация

- `useMemo` на тяжёлом расчёте фильтрации/сортировки/пагинации (`useCandidates`).
- `React.memo` на компонентах, получающих пропсы (карточки списка, строки виртуального списка).
- Виртуализация большого списка через `react-window`.
- Code-splitting: тяжёлый режим виртуализации и его данные (120 кандидатов, ~190 КБ) загружаются лениво через `React.lazy` + динамический `import()` — это убрало ~200 КБ из основного бандла.
- Cleanup в эффектах (`clearTimeout` в `useDebounce`, флаг отмены в загрузке большого списка) — нет утечек и обновлений размонтированных компонентов.

## Доступность (a11y)

- Строки таблицы доступны с клавиатуры (`role="button"`, `tabIndex`, обработка Enter/Space, focus-ring).
- `aria-label` на поле поиска, селекте сортировки и селекте смены статуса.

## Что можно развить дальше

**Данные и сеть**
- Подключение реального API с React Query вместо ручной загрузки в сторе — кэш, инвалидация, фоновые рефетчи и оптимистичные апдейты «из коробки».
- `keepPreviousData` при пагинации — старые строки остаются на экране во время загрузки новой страницы, без мерцания.
- Виртуализация с сохранением фильтров/сортировки (сейчас режим виртуализации — чистый список без фильтров, как демонстрация техники).

**Функциональность HR-домена**
- Гранулярный RBAC: разные права для ролей рекрутёр / хайринг-менеджер / интервьюер / админ (хелперы вида `canEditStatus()`, `availableActionsFor(role)`).
- Массовые операции над выбранными кандидатами (перевод этапа, тэги, рассылка) — чекбоксы с indeterminate-состоянием и единым хуком выбора.
- Bulk-импорт пула кандидатов из Excel/CSV по шаблону.
- HR-аналитика: воронка найма, time-to-hire, источники кандидатов (графики на Recharts).

**Архитектура и качество**
- Переход на FSD при росте проекта (`app / pages / widgets / features / entities / shared`) — когда кандидаты/вакансии/интервью станут отдельными сущностями со своими фичами.
- Рантайм-валидация ответов реального API через Zod (схемы + инференс типов).
