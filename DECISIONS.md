# Принятые решения

Документ с обоснованием технических решений и контекстом разработки.

## Процесс

- **Старт разработки:** 12 июня, 18:00.
- **История коммитов** отражает ход работы: задания закрывались последовательно (список → карточка → статусы → тесты → оптимизация), с отдельными коммитами на рефакторинги и правки UI. Сообщения коммитов в формате Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `style:`).

## Выбор инструментов

### Zustand вместо Redux Toolkit
ТЗ допускает оба. Выбрал Zustand: меньше бойлерплейта, нет провайдеров и экшен-криэйторов, удобные селекторы. Состояние разделено на два независимых стора — `candidatesStore` (данные + загрузка + смена статуса) и `filtersStore` (поиск, вердикт, сортировка, страница). Везде используются селекторы (`useStore(s => s.field)`), чтобы компоненты не перерисовывались на несвязанные изменения.

### React Router v7 — `createBrowserRouter`
Современный data-router API v7. Маршруты вынесены в отдельный модуль `routes/`.

### sonner для toast
Лёгкая и современная библиотека уведомлений, минимум настройки. Используется для обратной связи при смене статуса.

### react-window для виртуализации
Стабильная v1 с `FixedSizeList`. Строки фиксированной высоты → расчёт видимого окна без измерения DOM. Применена в отдельном режиме на 120 кандидатах.

### Tailwind CSS v4
Быстрая разработка консистентного UI. Утилита `cn()` (clsx + tailwind-merge) для безопасного объединения классов.

## Структура проекта

Использована простая структура из ТЗ (`components / pages / hooks / store / services / ...`), а **не FSD**.

**Про FSD:** знаком с методологией (слои `app / pages / widgets / features / entities / shared`, правило однонаправленных импортов). Применил бы её в крупном продакшен-проекте с несколькими командами и растущим числом фич, где важна жёсткая изоляция слоёв. Для тестового задания такого объёма FSD — оверинжиниринг, поэтому выбрал структуру, заданную в ТЗ. Доменные сущности при этом вынесены отдельно (`constants/`, `types/`) как единый источник правды — это частично перекликается со слоем `entities`.

## Что намеренно не использовал

### Zod
Знаком с библиотекой (схемы валидации, инференс типов из схем). Применил бы для валидации форм и проверки данных с реального API на рантайме. В этом проекте нет ни форм со сложной валидацией, ни реального бэкенда — данные приходят из статичных моков и уже типизированы, поэтому рантайм-валидация была бы избыточной.

### React Query (TanStack Query)
Знаком (кэширование запросов, инвалидация, фоновые рефетчи, оптимистичные апдейты «из коробки»). Применил бы при работе с реальным API — это убрало бы ручное управление loading/error в сторе. Здесь данные мок-статичные и грузятся один раз, поэтому простого Zustand-стора с ручной загрузкой достаточно; тащить React Query ради одного запроса не оправдано.

## Оптимизация

- `useMemo` на тяжёлом расчёте фильтрации/сортировки/пагинации (`useCandidates`).
- `React.memo` на компонентах, получающих пропсы (карточки списка, строки виртуального списка).
- Виртуализация большого списка через `react-window`.
- Code-splitting: тяжёлый режим виртуализации и его данные (120 кандидатов, ~190 КБ) загружаются лениво через `React.lazy` + динамический `import()` — это убрало ~200 КБ из основного бандла.
- Cleanup в эффектах (`clearTimeout` в `useDebounce`, флаг отмены в загрузке большого списка) — нет утечек и обновлений размонтированных компонентов.

## Доступность (a11y)

- Строки таблицы доступны с клавиатуры (`role="button"`, `tabIndex`, обработка Enter/Space, focus-ring).
- `aria-label` на поле поиска, селекте сортировки и селекте смены статуса.

## Что можно развить дальше
